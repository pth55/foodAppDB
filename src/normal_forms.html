NORMALIZATION:

  1.  First Normal Form (1NF):
  • Each table already has a primary key, ensuring uniqueness.
  • Every column contains atomic values, without repeating groups or arrays.
  • Each cell has a single value.
  • Each column has a unique name.
  • Based on the above schemas All tables adhere to 1NF.

 

  1. Second Normal Form (2NF):
  • A table is in 2NF if it's in 1NF and all non-key attributes are fully functionally dependent on the entire primary key.
  • In the above tables, all non-key attributes are fully functionally dependent on the entire primary key, so they already satisfy 2NF.

 

  1. Third Normal Form (3NF):
  • In Third Normal Form (3NF), a table is in 3NF if it's in Second Normal Form (2NF) and every non-key attribute is non-transitively dependent on the primary key. This means that every non-key column should depend only on the primary key, and not on any other non-key column.
  • If we Look at Address table, in that State, City, Street, and pin_code are functionally dependent on addr_id, but they are also functionally dependent on each other. This creates a transitive dependency, violating 3NF.
  • To resolve this, we can split the Address table into two tables:

Address_Details:

CREATE TABLE Address_Details (

    addr_id INT PRIMARY KEY,

    State VARCHAR(20),

    City VARCHAR(20),

    Street VARCHAR(20),

    pin_code INT

);

 

      User_Address:

CREATE TABLE User_Address (

    user_id INT,

    addr_id INT,

    PRIMARY KEY (user_id, addr_id),

    FOREIGN KEY (user_id) REFERENCES Users(id),

    FOREIGN KEY (addr_id) REFERENCES Address_Details(addr_id)

);

 

  1. Boyce-Codd Normal Form (BCNF):
  • In Boyce-Codd Normal Form (BCNF), a table is in BCNF if every determinant is a candidate key. This means that every non-trivial functional dependency in the table has a candidate key as its determinant.
  • To ensure BCNF, we need to verify that every determinant in each table is a candidate key. A candidate key is a minimal superkey for a table. If any determinant is not a candidate key, we need to further decompose the table.
  • After splitting the Address table as mentioned above, both Address_Details and User_Address tables satisfy BCNF.
